# 前端开发规范

适用范围：`frontend/` 下的页面、布局、组件、样式、路由、权限、SEO、测试脚本与构建配置。

本文是当前系统前端开发的统一约束，覆盖设计、实现、交互、性能、可维护性和交付要求。后续新增页面或重构页面，必须以本文为准，不再允许页面各自建立私有规范。

## 1. 目标

1. 建立一套适用于“公文写作系统”的正式、稳定、可维护的前端标准，而不是继续沿用模板式页面拼装。
2. 保证落地页、登录页、写作工作区、管理后台在视觉语言、编码方式、权限语义和交互质量上统一。
3. 让页面开发默认复用现有公共骨架与组件，减少重复实现、样式分叉和权限遗漏。
4. 把前端交付标准前置到开发阶段，确保类型检查、权限回归、视觉回归和生产构建都可重复执行。

## 2. 当前系统基线

### 2.1 技术栈

1. 框架：Vue 3 + TypeScript + `<script setup>`。
2. 构建：Vite 8。
3. 路由：Vue Router 4。
4. 状态：Pinia。
5. UI 基础：Element Plus + UnoCSS。
6. 编辑器：TinyMCE。
7. 网络：Axios。
8. 校验：Zod + Vee Validate。
9. 测试与回归：Playwright、权限回归脚本、`vue-tsc`、ESLint、Stylelint。

### 2.2 当前信息架构

1. 公共入口：落地页 `/`、登录页 `/login`。
2. 写作工作区：
   - `/chat` 写作会话
   - `/chat/:sessionId` 写作工作台
   - `/materials` 素材管理
   - `/book-learning` 书籍学习
   - `/history` 导出历史
   - `/settings` 写作偏好
3. 系统管理区：
   - `/admin/overview` 管理概览
   - `/admin/accounts` 账户与用户
   - `/admin/roles` 角色与权限
4. 后端接口、权限编码、RBAC 语义、路由路径为既定约束。前端可以重做布局和表达，但不能私自改动这些契约。

### 2.3 当前产品定位

1. 产品名统一为“公文写作系统”。
2. 前端视觉方向统一为“稳重公务风”：克制、正式、清晰、可治理。
3. 写作页是生产界面，不是聊天玩具页面。
4. 管理页是可信边界的一部分，不是附属配置页。
5. 落地页是正式对外页面，不是内部管理系统首页的简化版。

## 3. 开发原则

### 3.1 架构原则

1. 优先复用，而不是复制页面结构和样式。
2. 先确定页面属于哪一层，再开始实现。
3. 页面组件负责组织信息，不直接承担过多数据加工逻辑。
4. 数据加工、权限判断、展示格式化、SEO 同步必须落到明确层次，避免散落在模板中。
5. 新能力优先接入现有壳层、公共组件和权限体系，不新增第二套体系。

### 3.2 UI 原则

1. 页面首先服务任务，不服务装饰。
2. 信息层级靠排版、留白、边框、分区和状态语义表达，不靠高饱和颜色堆砌。
3. Writer 模块遵循“先会话，后工作台”。
4. Admin 模块遵循“表格优先，卡片辅助”。
5. 顶栏不承担页面标题职责，页面主标题只允许出现在 `PageHeader`。

### 3.3 工程原则

1. 默认使用 TypeScript 严格类型，不接受 `any` 扩散。
2. 业务代码必须可拆分、可测试、可复用。
3. 禁止用一次性写法绕过规范，特别是样式覆盖、权限显隐和第三方组件 hack。
4. 新增依赖必须说明用途、收益和体积影响。
5. 所有交付必须经过静态检查和至少一轮功能回归。

## 4. 目录与分层规范

### 4.1 目录职责

1. `src/layouts/`
   - 仅放应用壳层、导航结构与布局级逻辑。
   - 不写页面级业务逻辑。
2. `src/components/`
   - 放全局可复用基础组件或通用业务组件。
   - 只要出现第二个页面也要用到的结构，就优先考虑进入这里。
3. `src/views/`
   - 放页面入口。
   - `views/writer/` 只承载写作域页面。
   - `views/admin/` 只承载管理域页面。
4. `src/api/`
   - 放接口模块和请求封装。
   - 不在页面中直接手写原始请求地址。
5. `src/router/`
   - 放路由配置、路由守卫和权限路由逻辑。
6. `src/utils/`
   - 放无状态工具、格式化器、SEO 工具、常量和通用函数。
7. `src/assets/styles/`
   - 放全局 token、基础样式和系统级覆盖。
8. `scripts/`
   - 放权限回归、视觉回归、构建辅助脚本。

### 4.2 分层边界

1. 页面层只负责：
   - 页面结构
   - 页面级数据流编排
   - 组件组装
2. 组合逻辑层 `useXxx.ts` 只负责：
   - 状态组织
   - 事件封装
   - 接口编排
   - 视图无关的数据派生
3. 工具层只负责：
   - 纯函数
   - 格式化
   - SEO/head 更新
   - 无副作用或可控副作用的通用逻辑
4. 公共组件层只负责：
   - 通用展示逻辑
   - 明确 props / slots 接口
   - 不耦合特定页面 API

### 4.3 命名约定

1. Vue 组件文件使用 `PascalCase.vue`。
2. 组合函数使用 `useXxx.ts`。
3. 工具文件使用小写语义名，如 `seo.ts`、`permission.ts`。
4. 样式 class 使用语义化块名，不使用 `.box1`、`.wrap2`、`.left-area` 这类不可维护命名。
5. 常量命名使用大写蛇形，如 `LAST_SESSION_STORAGE_KEY`。
6. 路由 name 使用稳定语义名，不随页面标题改动。

## 5. 页面与组件结构规范

### 5.1 页面骨架

所有业务页默认采用以下结构：

1. `PageShell`
2. `PageHeader`
3. 主体内容区

规则：

1. 页面主标题只允许由 `PageHeader` 提供。
2. 顶栏仅展示当前分区、账户菜单和必要全局操作，不重复输出页面 H1。
3. 页面正文不允许再套第二层“伪页面壳”。

### 5.2 必须优先复用的公共组件

1. `PageShell`
2. `PageHeader`
3. `ActionBar`
4. `PanelCard`
5. `DataTableShell`
6. `EmptyState`
7. `StatusBadge`
8. `MetaTag`

规则：

1. 状态使用 `StatusBadge`，只表达运行状态、结果状态、启用状态。
2. 类型、权限、附加属性使用 `MetaTag`，不得与状态标签混用。
3. 表格页默认先用 `DataTableShell`，不要在每个页面自写表格外壳。
4. 页面顶部操作条默认用 `ActionBar`，不要重复写按钮分组结构。

### 5.3 组件拆分阈值

出现以下任一情况必须拆分：

1. 单文件页面模板、脚本、样式任一部分明显超过可读阈值。
2. 同一页面同时承载列表、详情、弹窗、表单、权限显隐、表格列格式化多种职责。
3. 同类结构在两个以上页面重复出现。
4. 样式超过 250 行且无法一眼看清模块边界。

### 5.4 模板规范

1. 模板只写结构与简单状态判断。
2. 不在模板中写复杂三元表达式和长链式判断。
3. 重复逻辑必须提取到 `computed`、工具函数或子组件。
4. 禁止在模板中直接拼接显示文案、权限编码、颜色值和大量类名分支。
5. 默认使用插槽扩展组件，而不是复制一份组件代码改细节。

## 6. Writer 模块规范

### 6.1 信息架构

1. 先会话列表，再进入工作台。
2. 会话页负责：
   - 新建会话
   - 筛选与搜索会话
   - 进入最近任务
3. 工作台页负责：
   - AI 对话
   - 工作流步骤
   - 正文编辑
   - 引用与导出
4. 不允许把“会话选择”和“全功能工作台”再次耦合成同一个重页面入口。

### 6.2 工作台布局

1. 桌面端默认保持“会话导航 / 主工作区 / 编辑器或信息面板”的明确主次关系。
2. 主写作区必须是视觉第一焦点。
3. 编辑器是持续可见的生产面板，不是独立弹窗。
4. 移动端允许切换视图，但状态必须明确，不得让用户失去当前任务上下文。

### 6.3 写作状态表达

1. 工作流步骤必须可扫描，不能只靠颜色点区分状态。
2. 用户消息、AI 消息、系统过程消息要有稳定结构和视觉区分。
3. 会话状态、生成中状态、导出状态、引用状态必须可观测。
4. 写作页文案统一使用中文，避免中英混杂和技术占位词暴露给用户。

### 6.4 编辑器规范

1. TinyMCE 继续作为唯一富文本编辑器方案。
2. 编辑器工具栏样式必须并入系统视觉，不可呈现第三方原生风格割裂感。
3. 编辑器头部只放与正文生产直接相关的状态和动作。
4. 编辑器内部样式覆盖限制在组件内部，不污染全局。
5. 与公文排版相关的字体、行高、页边距只允许出现在编辑器正文区域。

## 7. Admin 模块规范

### 7.1 总体原则

1. 管理页以数据可读、操作可控、权限明确为核心。
2. 表格是主角，卡片只做摘要和辅助说明。
3. 所有操作显隐必须同时满足前端权限判断和后端权限校验。

### 7.2 账户与用户页

1. 优先呈现资源列表、用户表、邀请码表。
2. 摘要卡片不得压过主要表格信息。
3. 行内操作统一用文本按钮组或轻量操作组，不用大面积按钮堆叠。

### 7.3 角色与权限页

1. 角色列表应优先支持扫描、定位和编辑。
2. 权限说明按功能域分组展示，不做均质平铺。
3. 权限标签与状态标签必须分开表达。
4. 删除、停用等破坏性动作必须二次确认。

## 8. 路由、权限与 RBAC 规范

### 8.1 路由元信息

每个路由必须定义清晰的 `meta`：

1. `title`
2. 必要时的 `icon`
3. 是否公开 `public`
4. 访问权限 `auth`

规则：

1. 页面标题来源于路由 `meta.title` 和 `PageHeader`，不得出现多套标题来源。
2. 新增管理页必须同时补菜单显隐、路由守卫和页面级按钮权限。
3. 旧路径兼容只能在路由层做，不得在页面内部混写跳转兼容逻辑。

### 8.2 权限判断

1. 菜单显隐由统一权限判断控制。
2. 路由访问由守卫控制。
3. 按钮显隐由页面权限状态控制。
4. 前端权限判断只负责 UX，不替代后端授权。
5. 禁止在页面局部写死权限字符串分支而不复用统一工具。

### 8.3 账户边界

1. 前端必须尊重账户级资源隔离语义。
2. 页面不得默认展示跨账户信息。
3. 需要跨账户切换时，必须在 UI 上明确当前账户上下文。
4. 与账户、用户、角色、邀请码相关的页面必须保持术语一致。

## 9. 样式与设计系统规范

### 9.1 真源

1. 视觉 token 以 `frontend/src/assets/styles/globals.css` 为真源。
2. 页面和组件只能消费 token，不允许重新发明第二套主题变量。
3. 若确需新增 token，必须先补到全局，再进入页面使用。

### 9.2 视觉方向

1. 主视觉：黑、白、灰、暖灰。
2. 整体气质：正式、安定、克制、可信。
3. 禁止重新引入紫色系、赛博发光、玻璃拟态堆叠、强渐变装饰等与系统气质冲突的表达。
4. 允许少量暖灰背景和低饱和分层，但必须服务结构表达。

### 9.3 颜色规范

1. 禁止在页面中直接写十六进制颜色、`rgb()`、`rgba()`，token 文件除外。
2. 状态色只用于状态表达，不用于页面主视觉。
3. Element Plus 变量覆盖以系统级覆盖为主，页面内禁止临时修改主题色板。

### 9.4 排版规范

1. 中文 UI 字体栈统一使用全局定义，不允许页面私配字体。
2. 页面标题、副标题、表头、正文、辅助说明必须使用固定字阶。
3. 12px 仅用于辅助说明、状态补充和技术标识。
4. 长段文案必须控制行宽，避免桌面端整屏拉满。

### 9.5 间距与圆角

1. 间距以 4 的倍数为基础。
2. 卡片和表单内部间距优先使用 `16 / 20 / 24`。
3. 圆角统一使用系统圆角 token，禁止某些页面大圆角、某些页面直角混用。
4. 阴影只表达层次，不表达“漂浮感”。

### 9.6 按钮规范

1. 主按钮：黑底白字。
2. 次按钮：白底深边。
3. 文本按钮：仅用于轻操作和行内操作。
4. 危险按钮：只用于破坏性动作。
5. 同一区域只允许一个主按钮。

### 9.7 表格规范

1. 管理页和数据页统一走“表格优先”。
2. 行高、表头、空状态、行内操作、tooltip 规则统一。
3. 长文本必须有省略和悬浮提示策略。
4. 表格外层必须可区分为完整信息区，不允许裸表格直接落在页面背景上。

## 10. 响应式与兼容性规范

### 10.1 响应式要求

1. 所有页面必须同时覆盖桌面端和移动端可用性。
2. 移动端优先保障：导航、主操作、表单提交、会话切换。
3. 桌面端优先保障：阅读宽度、信息密度、三栏或双栏关系稳定。
4. 侧栏在桌面端固定，不随内容区滚动。

### 10.2 兼容性要求

1. 必须保证 Chrome、Edge 可用。
2. Safari 和移动端浏览器在无 `backdrop-filter`、`mask-image`、`view-transition` 时仍可正常操作。
3. 增强特性必须有降级策略，不能把核心信息放在只在新浏览器可见的效果上。

### 10.3 视觉回归要求

1. 不能只依赖全页长截图判断布局正确。
2. 必须增加视口内截图、关键区块截图或 DOM 尺寸断言，避免宽度异常被长截图掩盖。
3. 关键页面至少覆盖桌面端和移动端两个视口。

## 11. 数据流、状态与接口规范

### 11.1 状态使用边界

1. Pinia 只用于跨页面、跨模块共享状态。
2. 页面局部状态优先放在页面或 `useXxx.ts` 中。
3. 不允许把一次性弹窗状态、局部 loading、局部筛选塞进全局 store。

### 11.2 接口使用规范

1. 所有接口走 `src/api/modules/` 封装。
2. 页面不得直接拼装请求 URL。
3. 响应数据如需展示格式化，优先在工具层或页面组合逻辑中处理，不在模板内反复格式化。
4. 错误提示统一使用中文面向用户文案，不暴露内部路径、堆栈和原始后端异常。

### 11.3 展示格式化

1. 时间、状态、大小、统计文案、标签文本等需要统一格式化。
2. 同一类字段不能在不同页面出现不同文案和格式。
3. 若一个格式化逻辑在两个页面出现，应提取为工具函数或展示组件。

## 12. SEO 与公开页面规范

### 12.1 适用页面

1. 落地页必须具备正式 SEO 能力。
2. 登录页、404 页等公开页面应至少具备正确标题。
3. 内部工作区页面无需堆叠营销型 SEO，但标题必须准确。

### 12.2 实现规范

1. 静态基础 meta 放在 `frontend/index.html`。
2. 运行时页面级 head 同步统一使用 `src/utils/seo.ts`。
3. 公开页面必须设置：
   - `title`
   - `description`
   - `canonical`
   - `og:title`
   - `og:description`
   - `og:url`
   - `og:image`
   - `twitter:card`
   - `twitter:title`
   - `twitter:description`
   - `twitter:image`
4. `VITE_SITE_URL` 是公开链接真源，部署时必须配置真实域名。
5. 分享图、图标、manifest 必须放在 `frontend/public/` 并与当前品牌一致。

## 13. 文案、编码与文本质量规范

1. 对外产品文案统一使用中文。
2. 产品名称统一为“公文写作系统”，不得混用旧名称。
3. 禁止出现乱码、问号替代字符、异常引号、BOM 导致的文本污染。
4. 所有前端源码、文档、配置文件统一使用 UTF-8 无 BOM。
5. 中文文案必须使用正常汉字和中文标点，不允许把控制台乱码复制回源码。
6. 页面中的技术性错误文案要翻译成可理解的用户语言。

## 14. 性能与构建规范

### 14.1 代码组织

1. 路由组件默认异步加载。
2. 重依赖必须按业务边界拆分，避免首页无关资源进入首屏。
3. 大体积第三方依赖必须尽量限制在使用页面内。
4. 离线图标数据、编辑器资源等必须按需使用，避免无差别打包。

### 14.2 样式性能

1. 避免深层嵌套选择器。
2. 避免大量页面私有 token 和局部主题覆盖。
3. `:deep()` 仅允许第三方组件覆盖场景使用。
4. `!important` 仅允许解决第三方不可控样式冲突。

### 14.3 交互性能

1. 避免匿名全局事件监听长期挂载。
2. 监听器、定时器、观察器必须在组件卸载时清理。
3. 列表和表格状态更新必须尽量局部化，避免整个页面重复渲染。

## 15. 测试与交付规范

### 15.1 基础校验

提交前至少执行：

1. `pnpm -C frontend exec vue-tsc --noEmit`
2. `pnpm -C frontend exec eslint .`
3. `pnpm -C frontend exec stylelint "src/**/*.{css,scss,vue}"`
4. `pnpm -C frontend run test:permissions`
5. `pnpm -C frontend run --config.verify-deps-before-run=false build`

### 15.2 页面回归

涉及 Writer 时至少回归：

1. 会话列表进入
2. 新建、切换、删除会话
3. 写作工作台加载
4. 发送、停止、导出、编辑器显示
5. 素材管理关键流程
6. 书籍学习扫描、导入、进度

涉及 Admin 时至少回归：

1. 管理菜单显隐
2. 路由守卫
3. 账户与用户操作
4. 角色与权限操作
5. 按钮权限显隐

涉及落地页或公开页面时至少回归：

1. 桌面端首屏
2. 移动端首屏
3. SEO meta 输出
4. CTA 路径跳转
5. 分享图、图标、标题是否正确

### 15.3 Playwright 规范

1. 重要流程必须优先做点击式真实回归。
2. 截图对比只作为补充，不替代行为验证。
3. 关键布局同时检查：
   - 当前视口截图
   - 关键容器尺寸
   - 交互后状态变化
4. 新增页面或重大重构时必须补一组稳定选择器，避免回归脚本脆弱。

## 16. 禁止项

以下行为默认禁止：

1. 在页面中直接写硬编码颜色。
2. 在模板中写大段业务逻辑和复杂格式化。
3. 在多个页面复制同一套标题区、表格壳层或空状态结构。
4. 页面私自定义第二套按钮语义、状态标签语义或表格样式。
5. 通过隐藏 DOM 代替真正的权限控制。
6. 为了赶进度直接跳过类型检查、权限回归和构建验证。
7. 把乱码文案、控制台异常文案或英文占位词直接暴露给用户。
8. 在未评估的情况下引入新的 UI 框架或状态管理方案。

## 17. 交付清单

每次前端改动合并前，必须自查：

1. 是否复用了现有壳层和公共组件。
2. 是否引入了新的硬编码样式和值。
3. 是否满足当前模块的信息架构原则。
4. 是否补齐权限显隐、路由守卫和后端契约对齐。
5. 是否修复了桌面端和移动端布局。
6. 是否执行了静态检查、权限回归和生产构建。
7. 是否检查了中文文案、编码、引号和乱码问题。
8. 如果是公开页面，是否补齐 SEO / 分享信息。

## 18. 后续维护原则

1. 本规范优先级高于页面历史写法。
2. 发现现有页面不符合规范时，应优先修正，而不是照着历史问题继续复制。
3. 新的公共模式一旦稳定，应回收进公共组件或全局 token，而不是停留在单页内部。
4. 规范更新必须基于当前真实系统，而不是抽象模板经验。

# 前端设计规范

适用范围：`frontend/src/` 下的页面、布局、组件、样式文件与设计 token。

## 1. 目标

1. 保持全站黑白灰主视觉，避免页面级各自定义主题色。
2. 统一页面骨架、卡片、表格、表单、按钮、空状态与弹窗样式。
3. 样式以全局 token 为真源，页面样式只负责组合，不重复定义视觉规则。
4. 保证桌面端与移动端都可用，并控制样式复杂度，方便后续持续扩展。

## 2. 设计原则

1. 风格关键词：克制、清晰、正式、稳定。
2. 主色策略：仅使用黑、白、灰作为系统主视觉；状态色仅用于状态标签和反馈，不作为页面主色。
3. 信息层级：通过字号、字重、留白和边框层次表达，不依赖高饱和颜色。
4. 交互反馈：优先使用 hover、active、focus-visible、阴影和边框变化，不叠加多套特效。

## 3. Token 规范

### 3.1 颜色

以 `frontend/src/assets/styles/globals.css` 中的 token 为真源，统一使用：

- 基础色：`--w-color-black`、`--w-color-white`
- 中性色：`--w-gray-50` 到 `--w-gray-900`
- 阴影：`--w-shadow-xs`、`--w-shadow-sm`、`--w-shadow-md`
- 圆角：`--w-radius-sm`、`--w-radius-md`、`--w-radius-lg`
- 按钮：`--w-btn-primary-*`、`--w-btn-secondary-*`

规则：

1. 页面和组件内禁止直接写十六进制颜色、`rgb()`、`rgba()`，除非是在 token 定义文件内。
2. 不允许在单个页面局部重写 Element Plus 主题色变量，除非封装为全局语义 token。
3. 空状态图标、弱提示文字、卡片边框统一走 token，不写页面私有色值。

### 3.2 间距

间距统一采用 4 的倍数，推荐序列：`4 / 8 / 12 / 16 / 24 / 32 / 40`。

规则：

1. 页面外边距默认 `24px`，移动端默认 `16px`。
2. 卡片内部默认 `16px` 或 `20px`。
3. 同层级组件间距保持一致，不在同一页面混用 `10 / 14 / 18 / 22` 这类无规律值。

### 3.3 圆角与边框

1. 输入框、选择器、普通卡片默认使用 `--w-radius-md`。
2. 气泡、大面板、编辑器容器可使用 `--w-radius-lg`。
3. 边框统一使用 `--w-gray-200` 或 `--w-gray-300`，避免同页出现多套边框灰度。

### 3.4 字体与排版

1. 普通界面优先继承系统字体，不在页面层私自指定花哨字体。
2. 页面标题统一 22px / 700。
3. 页面副标题统一 13px / 常规字重 / 次级文字色。
4. 正文统一 14px~16px，说明文字统一 12px~13px。
5. 代码、权限编码等技术标识允许使用等宽字体。
6. 公文编辑器正文、标题字体可按公文规范单独定义，但只允许出现在编辑器内容区。

## 4. 组件规范

### 4.1 页面骨架

所有业务页应优先复用以下结构：

- `page-shell`：页面容器，负责 `padding`、背景、最外层布局
- `page-header`：标题区，包含标题、副标题、右侧操作区
- `page-content`：主体内容区

规则：

1. 不允许每个页面重新定义一套 `page-title/page-subtitle/page-header` 样式。
2. 页面背景统一使用全局页面背景 token；若使用渐变，应抽成公共页面背景样式。

### 4.2 按钮

1. 主按钮：黑底白字。
2. 次按钮：白底黑字，使用标准边框。
3. 文字按钮：仅用于次级操作，不得承担主流程提交动作。
4. 危险按钮：仅用于删除、清空等破坏性动作。
5. 同一操作区最多只允许 1 个主按钮。

### 4.3 卡片与表格

1. 卡片统一使用白底、浅灰边框、轻阴影。
2. 表格外层统一包裹卡片或表格壳层，不允许某些页面有边框、某些页面完全裸奔。
3. 空状态统一包含：图标、提示文案、可选引导动作。

### 4.4 表单与弹窗

1. 输入框、选择器、文本域圆角统一。
2. 弹窗 footer 按钮顺序统一为：次按钮在左，主按钮在右。
3. 表单内联样式应提取为 class，避免模板内写 `style="width: ..."`。

### 4.5 编辑器与第三方组件

1. 第三方组件样式覆盖只允许放在组件自己的样式块内。
2. `:deep()` 仅用于无法通过 props 或 class 解决的第三方内部结构覆盖。
3. `!important` 仅用于覆盖第三方内联样式或不可控优先级，不得作为常规方案。

## 5. 响应式规范

1. 页面默认支持桌面端与移动端两套布局。
2. 断点优先使用 `768px` 作为首个移动断点。
3. 移动端应优先保证可读性和可点击性，而不是强行保留桌面布局。
4. 浮动按钮、抽屉、遮罩层必须明确层级关系和关闭路径。

## 6. 兼容性规范

1. 构建链已启用 `autoprefixer` 和 `@vitejs/plugin-legacy`，但使用新特性时仍需提供降级方案。
2. 对 `backdrop-filter`、`mask-image`、`view-transition`、自定义滚动条这类增强特性，必须保证在不支持时页面仍可正常使用。
3. 不依赖 `-webkit-` 私有属性作为唯一实现。

## 7. 样式代码规范

1. 样式文件统一使用 token，不直接写视觉值。
2. 样式属性顺序遵循 stylelint 规则。
3. 页面样式超过 250 行时，应优先拆分为子组件或独立样式文件。
4. 同一视觉模式出现 3 次及以上时，必须抽为公共 class 或公共组件。
5. 中文产品界面文案保持统一语言，不混用整页英文标题和中文页面。
6. 注释只保留必要说明，禁止保留乱码注释或失真文案。

## 8. 审查清单

提交前至少检查：

1. 是否复用了全局 token，而不是新增页面私有颜色。
2. 是否出现重复的 `page-header/page-title/page-subtitle/card/table/empty-state` 样式。
3. 是否存在不必要的 `:deep()`、`!important`、内联 `style`。
4. 是否补齐移动端样式。
5. 是否通过 `vue-tsc`。
6. 是否通过 `stylelint`，或明确记录例外原因。

# 项目通用开发规范

适用范围：`backend/`、`frontend/`、`scripts/`、根目录配置、部署脚本与项目文档。

优先级说明：
1. 仓库根目录的 `AGENTS.md` 与显式任务要求优先级最高。
2. 本文是当前项目的通用总纲，负责约束跨前后端、跨目录、跨交付链路的共同行为。
3. 后端细则以 [backend-development-standard.md](./backend-development-standard.md) 为准。
4. 前端细则以 [frontend-design-standard.md](./frontend-design-standard.md) 为准。
5. 若专项规范与本文冲突，在对应作用域内以专项规范为准；未覆盖部分回退到本文。

## 1. 项目现状与真源

### 1.1 当前项目定位

1. 产品名统一为“公文写作系统”。
2. 当前系统不是演示项目，而是包含正式工作区、管理区、书籍学习、素材管理、权限治理和导出链路的完整业务系统。
3. 任何新增能力都必须同时考虑：功能正确性、账户隔离、权限边界、接口契约、UI 一致性、部署可落地性。

### 1.2 当前架构真源

1. 前端：`Vue 3 + TypeScript + Vite + Element Plus + Pinia + TinyMCE`。
2. 后端：`FastAPI + SQLAlchemy 2.x + Alembic + Pydantic Settings`。
3. 检索与知识增强：`OpenViking`。
4. 文档解析与风格学习链路包含：`PyPDF2`、`ebooklib`、`beautifulsoup4`、`pytesseract`、`pdf2image`、`jieba`。
5. 文档、权限、契约、构建与部署规则都必须围绕这套已落地技术栈制定，不允许跳出当前系统另起第二套体系。

### 1.3 真源边界

1. 后端是以下语义的唯一真源：
   - 权限校验
   - 账户隔离
   - 数据结构
   - 错误边界
   - 时间语义
   - SSE 事件结构
2. 前端是以下语义的唯一真源：
   - 页面信息架构
   - 交互流程组织
   - 视觉 token 与组件表达
   - 公共页面骨架与状态语义件
3. OpenAPI 是前后端接口类型同步的唯一上游，前端生成产物必须由后端导出结果驱动。
4. README、`.env.example`、部署脚本和专题规范文档是交付真源的一部分，不是事后补充材料。

## 2. 仓库结构与目录职责

### 2.1 顶层目录职责

1. `backend/`：后端应用、迁移、测试、依赖和运行入口。
2. `frontend/`：工作区、落地页、管理区、路由、状态管理、生成类型与前端脚本。
3. `docs/`：项目级规范、专项规范、审计与落地文档。
4. `scripts/`：OpenAPI 导出、编码检查、部署辅助、开发脚本。
5. `data/`：运行时数据目录，包括数据库、上传、导出、OpenViking 工作空间、书籍目录。
6. 根目录配置文件（如 `.env.example`、`docker-compose.yml`、`ov.conf.example`、`README.md`）属于正式交付面，修改时必须与实现同步。

### 2.2 职责边界

1. 不允许把后端业务规则塞进前端页面组件。
2. 不允许把前端展示拼装写回后端模型层。
3. 不允许在脚本目录复制一份生产逻辑，脚本应复用正式服务能力或明确只做导出、校验、初始化。
4. 不允许把临时排障代码、一次性兼容分支或人工迁移说明长期留在主干却不收口。

## 3. 通用开发原则

### 3.1 正确性优先

1. 先保证行为正确、边界清晰、错误可追踪，再考虑局部性能与代码压缩。
2. 涉及权限、数据删除、导出、后台任务、书籍导入、会话上下文、记忆写入的改动，默认视为高风险改动。
3. 任何会改变历史数据、索引、权限可见性或接口结构的改动，都必须补说明、补测试或补回归脚本。

### 3.2 单一真源

1. 同一业务规则只能有一个正式落点。
2. 当规则同时出现在前端、后端、脚本、文档中时，必须明确谁是源、谁是派生产物。
3. 典型真源示例：
   - 权限编码真源：后端持久化模型与权限服务
   - 接口类型真源：后端 OpenAPI
   - 页面视觉真源：全局 token 与公共基础组件
   - 数据库结构真源：Alembic migration

### 3.3 兼容策略受控

1. 默认不新增长期兼容分支，除非需求明确要求。
2. 一次性重构、替换或清库方案必须写清楚边界，不允许代码里同时背两套长期语义。
3. 若确实需要兼容，必须明确：兼容期限、退出条件、覆盖测试和清理时间点。

### 3.4 可观测与可回退

1. 新增长流程必须可观测，至少满足“状态可见、错误可追踪、结果可验证”三项。
2. 外部副作用默认可降级，但降级结果必须对日志和调用方可见。
3. 不允许以“先上了再看日志”为主要交付方式。

## 4. 跨前后端协作规范

### 4.1 接口契约

1. 后端新增、删除或修改接口时，必须同步审视：
   - `response_model`
   - serializer 输出
   - 错误响应字段
   - SSE 事件格式
2. 前端不得长期手写与后端脱节的接口类型。
3. 修改接口契约后必须执行：

```bash
pnpm -C frontend run generate:openapi
pnpm -C frontend run verify:openapi
```

4. 若生成类型发生变化，相关前端调用、页面类型和回归脚本必须一并更新。

### 4.2 权限与账户边界

1. 所有受保护资源以“后端校验成功”为准，前端权限控制仅承担体验层职责。
2. 菜单显隐、路由守卫、按钮显隐必须共用同一套权限语义，不允许页面私写特殊分支。
3. 除显式声明为全局共享的资源外，业务数据默认按账户隔离。
4. 账户隔离不只针对数据库记录，也包括：
   - OpenViking 资源命名空间
   - 记忆检索
   - 素材与风格分析结果
   - 会话与导出记录
5. 任何新增资源在设计时都必须先回答：是否按账户隔离、如何校验权限、前端哪些角色可见。

### 4.3 数据库与迁移

1. 数据库结构变更以 Alembic 为唯一真源。
2. 禁止继续依赖 `create_all()`、运行时 schema patch 或隐式补字段作为正式迁移方案。
3. 新增表、字段、索引、唯一约束后，必须提交对应 migration。
4. 若改动涉及初始化数据、管理员初始化、权限字典或邀请码策略，也要明确写入迁移或显式 bootstrap 逻辑，而不是藏在页面首次访问流程中。

### 4.4 时间、路径与文本

1. 数据库存储统一使用 UTC。
2. API 对外展示统一转换为上海时区语义，避免前端各页自行解释。
3. 运行路径必须来自配置项与统一路径解析逻辑，不允许硬编码开发机绝对路径。
4. 所有源码、配置、JSON、Markdown、脚本输出统一使用 UTF-8 无 BOM。
5. 在本仓库中使用 PowerShell 处理中文输出或文本文件前，必须先执行：

```powershell
. .\scripts\powershell_utf8.ps1
```

6. 禁止使用 `Set-Content`、`Add-Content`、`Out-File`、`>`、`>>` 直接写仓库文本文件。
7. 发现 BOM、乱码、失真中文、错误引号时，必须修根因，不得仅在展示层做掩盖。

### 4.5 错误处理与安全

1. 对外响应只暴露通用错误信息和必要的 `error_id`。
2. 禁止向前端返回绝对路径、内部异常堆栈、数据库内部错误、第三方 SDK 原始报错。
3. 日志必须能关联最小必要上下文，例如：`operation`、`account_id`、`user_id`、`task_id`、`error_id`。
4. 不得在日志、前端、示例配置或 README 中暴露真实密钥、密码、令牌、完整邀请码或生产内部地址。
5. 默认示例密钥不得允许生产环境直接启动。

## 5. 页面、接口与任务的实现约束

### 5.1 前端页面与交互

1. 页面必须接入现有壳层、公共基础组件、权限体系和设计 token，不允许页面私起一套视觉系统。
2. 大页面出现以下任一情况时必须拆分：
   - 模板和脚本难以阅读
   - 样式超过可维护阈值
   - 出现可复用的局部结构
3. 页面状态要区分：加载中、空状态、无权限、失败、处理中，不允许全部复用同一种占位文案。
4. 影响主路径体验的改动，应优先补 Playwright 点击流或截图回归。

### 5.2 后端接口与服务

1. 路由层负责校验、鉴权、状态码和响应返回，不负责长链路编排。
2. 服务层负责业务流程、后台任务、外部系统交互和降级策略。
3. 模型层只定义持久化结构，不承载接口拼装与业务分支。
4. 任何可复用输出结构都应下沉到 serializer 或 schema，而不是在路由里手拼字典。
5. `async` 路由不得直接执行重 CPU 或阻塞 I/O 逻辑。

### 5.3 后台任务与外部副作用

1. 后台任务必须具备可查询状态、错误记录和可恢复的最小能力。
2. 书籍导入、OCR、导出、外部同步、记忆写入等长流程必须限制并发，避免竞争和线程风暴。
3. 外部副作用失败时是否允许降级，必须在服务层明确，不得由页面自行猜测。
4. 不允许在应用启动时自动执行高成本导入、全量重建或未经确认的数据清理。

## 6. 文档、配置与部署协同规范

### 6.1 文档同步

1. 需求落地后，若影响对外认知，必须同步更新 README。
2. 影响研发协作的规则变更，必须同步更新 `docs/` 下对应规范文档。
3. 新增配置项、路径规则、启动方式、初始化步骤时，必须同步更新：
   - `.env.example`
   - `README.md`
   - 部署脚本或示例配置
4. 文档不能只描述“理想状态”，必须与当前主干实现一致。

### 6.2 配置治理

1. 配置项统一进入后端 `Settings` 或前端环境变量体系，不允许散落在代码常量中。
2. 所有新增环境变量都要给出默认值、示例值、用途说明和部署影响。
3. 示例配置文件应可直接作为起点使用，但必须显式提醒手工替换敏感值。
4. `ov.conf` 等运行配置示例应提交示范文件，不提交真实环境文件。

### 6.3 部署与运行脚本

1. 部署脚本默认面向 Linux + Docker 场景，同时保证本地开发链路可用。
2. 脚本要做到可重复执行，避免隐式状态依赖。
3. 脚本输出应说明关键结果和待手工处理项，不能只打印原始命令输出。

## 7. 测试与交付门禁

### 7.1 通用要求

1. 提交前至少完成本次改动直接相关的静态检查与回归。
2. 任何涉及权限、接口契约、导入任务、会话流、导出链路、布局结构的改动，都不能只靠肉眼检查。
3. 检查失败时优先修根因，不允许通过降低检查门槛绕过。

### 7.2 后端改动最小检查

```bash
python -m compileall backend/app backend/tests scripts
python -m pytest backend/tests/test_regressions.py -q
python scripts/check_text_encoding.py --all
```

涉及数据库结构变更时，还必须执行：

```bash
python -m alembic -c backend/alembic.ini upgrade head
python -m alembic -c backend/alembic.ini check
```

### 7.3 前端改动最小检查

```bash
pnpm -C frontend exec vue-tsc --noEmit
pnpm -C frontend exec eslint .
pnpm -C frontend exec stylelint "src/**/*.{css,scss,vue}"
pnpm -C frontend run test:permissions
pnpm -C frontend run --config.verify-deps-before-run=false build
```

### 7.4 契约与视觉回归

1. 改动后端接口、SSE 事件或 serializer 输出时，必须补跑 OpenAPI 同步检查。
2. 改动落地页、写作工作区、管理区、书籍学习页、主壳层布局时，应补跑 Playwright 页面回归。
3. 影响桌面端布局的视觉回归，不能只截全页图；必须包含贴近真实视口的截图或点击流验证。

## 8. 禁止实践

1. 禁止在导入模块时执行数据库写入、建表、恢复任务、目录清理等副作用。
2. 禁止为赶进度把业务规则复制到第二个入口再各自演化。
3. 禁止在前端以硬编码方式绕过真实权限。
4. 禁止在后端以“先返回成功，后台再说”为名掩盖核心写库失败。
5. 禁止提交 BOM 文件、乱码文案、失真中文、错误引号和无意义整文件格式噪音。
6. 禁止提交未纳管的新接口、新环境变量、新路由而不补文档和检查。
7. 禁止把历史临时分支、模板残留能力、未使用依赖和失效脚本长期留在主干不治理。
8. 禁止把生产配置、真实密钥、真实内部地址提交进仓库。

## 9. 推荐开发流程

1. 先确认需求影响哪些真源：数据、权限、接口、页面、文档、部署。
2. 再确定实现落点：前端、后端、脚本、迁移、文档各改什么，不要在错误层次下手。
3. 优先复用现有公共层：serializer、schema、service、PageShell、PageHeader、权限守卫、OpenAPI 生成链路。
4. 实现完成后同步更新文档、示例配置和回归脚本。
5. 提交前执行对应检查，并人工复核 diff，清理无意义噪音。
6. 若本次改动引入新的项目级约束，应同步更新本文；若只影响前端或后端专项规则，则更新对应专项规范。

## 10. 当前阶段的强制共识

1. RBAC 以后端持久化模型与权限服务为真源。
2. 资源、记忆与检索默认按账户隔离，显式共享资源除外。
3. 书籍学习是系统级知识增强能力，入口由页面、API、CLI 共用同一后端服务实现。
4. 数据库结构通过 Alembic 管理，不再回退到无迁移方案。
5. 会话、管理区、书籍学习、落地页都属于正式产品面，任何改动都必须按正式交付标准回归。
